OWASP MASVS v2.1.0

Mobile Application Security Verification Standard
8 Control Groups · 24 Controls

Overview

OWASP MASVS adalah standar industri untuk keamanan aplikasi mobile. Menyediakan set kontrol keamanan komprehensif untuk menilai keamanan mobile apps di berbagai platform (Android, iOS) dan skenario deployment (consumer, enterprise).

  • Testing Profiles: L1, L2, dan R (Resilience) — kini disebut "MAS Testing Profiles" di MASTG, aligned dengan NIST OSCAL
  • 8 Kategori: Storage, Crypto, Auth, Network, Platform, Code Quality, Resilience, Privacy
  • Digunakan oleh: Google (MASA/ADA), CREST, institusi pemerintah & edukasi
  • Pendamping: OWASP MASTG (testing guide) & MASWE (weakness enumeration)
MASVS-STORAGE Storage
2 controls ▶
Memastikan data sensitif (PII, credentials, secrets, API keys) yang disimpan lokal terlindungi dengan benar, baik di storage internal maupun publik. Juga mencakup pencegahan kebocoran data tidak sengaja via backups, logs, keyboard cache, dll.
MASVS-STORAGE-1 The app securely stores sensitive data.
Data sensitif dari user, backend, system services, atau app lain harus disimpan dengan proteksi yang memadai, terlepas dari lokasi penyimpanan (internal storage, public folders, dll).

MASVS-CRYPTO Cryptography
2 controls ▶
Memastikan app menggunakan cryptography sesuai best practices industri (NIST SP 800-175B, NIST SP 800-57). Mencakup pemilihan algoritma yang kuat, konfigurasi yang benar, dan manajemen key lifecycle (generation, storage, protection).
MASVS-CRYPTO-1 The app employs current strong cryptography and uses it according to industry best practices.
Penggunaan cryptographic libraries yang terbukti, pemilihan dan konfigurasi cryptographic primitives yang tepat, serta random number generator yang sesuai. Mengikuti standar eksternal seperti NIST.

Random number generation: SecureRandom (Android) vs SecRandomCopyBytes (iOS) 6. Common cryptographic configuration mistakes 7. MASTG test cases for verifying crypto configuration (Frida method tracing, static analysis) 8. Code examples in Kotlin and Swift
MASVS-CRYPTO-2 The app performs key management according to industry best practices.
Manajemen cryptographic keys sepanjang lifecycle: generation, storage, dan protection. Keys tidak boleh hardcoded, tidak boleh didapat dari stable device features, tidak boleh disimpan tanpa proteksi tambahan, dan tidak boleh disinkronisasi antar device untuk high-risk data.

MASVS-AUTH Authentication & Authorization
3 controls ▶
Memastikan app mengimplementasikan authentication dan authorization secara aman. Mencakup protokol remote auth (OAuth2, dll), local authentication (biometrics, PIN), dan additional authentication untuk sensitive operations. Enforcement harus di server side, tapi app harus mengikuti best practices client-side.
MASVS-AUTH-1 The app uses secure authentication and authorization protocols and follows the relevant best practices.
Penggunaan protokol auth yang aman (OAuth2, OpenID Connect) dengan implementasi yang benar di sisi mobile app. Termasuk token storage, token refresh, dan session management.

MASVS-NETWORK Network Communication
2 controls ▶
Memastikan app membuat koneksi yang aman dan terenkripsi untuk semua network communication. Mencakup TLS configuration yang benar dan certificate/public key pinning untuk endpoint yang dikuasai developer.
MASVS-NETWORK-1 The app secures all network traffic according to the current best practices.
Semua network traffic harus dienkripsi dan endpoint diautentikasi via TLS. Developer tidak boleh menonaktifkan platform secure defaults atau membypassnya via low-level APIs atau third-party libraries.

MASTG test cases: MITM with Burp Suite, nscurl for ATS verification, Frida TLS bypass detection 7. Code examples in Kotlin and Swift for secure TLS configuration
MASVS-NETWORK-2 The app performs identity pinning for all remote endpoints under the developer's control.
Certificate pinning atau public key pinning untuk memastikan hanya CA spesifik yang dipercaya, bukan semua default root CAs dari framework atau device.

MASVS-RESILIENCE Resilience Against Reverse Engineering & Tampering
4 controls ▶
Defense-in-depth measures: code obfuscation, anti-debugging, anti-tampering, dll. Meningkatkan resilience app terhadap reverse engineering dan client-side attacks. Catatan: ketiadaan measures ini tidak menyebabkan vulnerability — ini adalah proteksi tambahan yang harus dikombinasikan dengan kontrol MASVS lainnya.
MASVS-RESILIENCE-1 The app validates the integrity of the platform.
Memvalidasi bahwa OS tidak telah di-tamper (rooted/jailbroken). Platform yang telah di-compromise dapat menonaktifkan fitur keamanan, sehingga banyak kontrol MASVS yang bergantung pada platform security menjadi tidak reliable.
▶ Prompt for detailed info
Copy this prompt
Explain MASVS-RESILIENCE-1 in detail. Include: 1. Android root detection: SafetyNet/Play Integrity API, file-based checks, Magisk detection 2. iOS jailbreak detection: file existence checks, sandbox violation, fork() test 3. Google Play Integrity API: classic request, standard request, deviceIntegrity verdicts 4. Android Key Attestation for platform integrity verification 5. iOS DeviceCheck and App Attest 6. Known bypasses: Magisk Hide, Frida root detection bypass, objection 7. MASTG test cases: bypassing root/jailbreak detection (Frida, patching) 8. Code examples in Kotlin and Swift
MASVS-RESILIENCE-2 The app implements anti-tampering mechanisms.
Mencegah modifikasi kode dan resources app. Tanpa proteksi, attacker dapat dengan mudah menjalankan versi yang dimodifikasi (cheat, unlock premium features) atau mendistribusikan versi yang di-backdoor.

Play Integrity API for app integrity (appIntegrity verdict) 5. Anti-repackaging techniques 6. Known bypasses: APKTool repackage, Frida runtime patching 7. MASTG test cases for tampering verification 8. Code examples in Kotlin and Swift
MASVS-RESILIENCE-3 The app implements anti-static analysis mechanisms.
Mempersulit pemahaman internal app melalui static analysis. Code obfuscation, string encryption, dan teknik lain untuk menghambat reverse engineer memahami cara kerja app.

MASVS-PRIVACY Privacy
4 controls ▶
Kontrol privacy yang melengkapi (bukan menggantikan) assessment komprehensif seperti DPIA (GDPR). Mencakup data minimization, user identification prevention, transparency, dan user control over data. Tidak boleh menggantikan assessment oleh privacy experts.
MASVS-PRIVACY-1 The app minimizes access to sensitive data and resources.
App hanya boleh meminta akses ke data yang benar-benar dibutuhkan dengan informed consent dari user. Third-party SDK harus beroperasi berdasarkan user consent, bukan by default. End-to-end responsibility untuk data di supply chain SDK (SBOM alignment).

Third-party SDK consent enforcement: consent signals, SDK behavior control 5. SDK supply chain: SBOM, transitive data sharing risks 6. MASTG test cases for data minimization verification 7. Code examples in Kotlin and Swift for permission management
MASVS-PRIVACY-2 The app prevents identification of the user.
Melindungi identitas user melalui teknik unlinkability: data abstraction, anonymization, dan pseudonymization. Mencegah user identification dan tracking.

Data abstraction: minimizing PII in logs, analytics, crash reports 6. MASTG test cases for identification prevention 7. Code examples in Kotlin and Swift
MASVS-PRIVACY-3 The app is transparent about data collection and usage.
App harus transparan tentang data yang dikumpulkan dan bagaimana digunakan. Termasuk privacy policy, in-app disclosures, dan data collection notices.

User harus memiliki kontrol atas data mereka: mekanisme untuk manage, delete, modify data, dan change privacy settings (revoke consent). App harus re-prompt for consent saat membutuhkan lebih data dari yang awalnya dispecified.
